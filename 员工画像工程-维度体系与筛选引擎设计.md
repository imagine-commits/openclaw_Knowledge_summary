# 员工画像工程 — 维度体系 & 筛选引擎设计方案

> 版本：v1.0 | 日期：2026-05-05
> 背景：中国移动数智事业部（九天公司 | 九天研究院）

---

## 一、整体架构

```
数据源（花名册/绩效/培训/项目）
    ↓
员工基础数据池
    ↓
┌─────────────────────────────────────────┐
│  画像维度体系（6大类 N个维度）            │
│  每个员工 → 每个维度 → 分值              │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  筛选引擎 Skill                          │
│  输入：任务需求 + 维度权重 + 公式           │
│  输出：员工综合排名列表                   │
└─────────────────────────────────────────┘
```

## 二、维度体系（6大类）

### 大类 1：基础属性维度（静态档案）

| 维度名 | 说明 | 数据来源 | 计分方式 |
|--------|------|----------|----------|
| `age` | 年龄 | 花名册 | 数值 |
| `tenure` | 司龄（月） | 花名册 | 数值 |
| `education_level` | 学历层级 | 花名册 | 1=专科 2=本科 3=硕士 4=博士 |
| `position_level` | 岗级 | 花名册 | 数值 |
| `group` | 所属组 | 花名册 | 枚举 |
| `work_location` | 工作归属地 | 花名册 | 枚举 |
| `political_status` | 政治面貌 | 花名册 | 1=群众 2=团员 3=党员 |
| `is_youth` | 是否青年（<40岁） | 花名册 | 0/1 二值 |

---

| 维度名 | 说明 | 数据来源 | 计分方式 |
|--------|------|----------|----------|
| `tech_skill` | 技术能力 | 项目经历/技术栈数据 | 1-5 级 |
| `management_skill` | 管理能力 | 360评估/晋升记录 | 1-5 级 |
| `business_understanding` | 业务理解能力 | 业务线年限/项目经历 | 1-5 级 |
| `communication` | 沟通能力 | 360评估 | 1-5 级 |
| `innovation` | 创新能力 | 项目成果/专利 | 1-5 级 |
| `learning_ability` | 学习能力 | 培训记录/学习时长 | 1-5 级 |
| `problem_solving` | 问题解决能力 | 360评估/项目复盘 | 1-5 级 |

---

### 大类 3：绩效维度

| 维度名 | 说明 | 数据来源 | 计分方式 |
|--------|------|----------|----------|
| `performance_score` | 绩效分 | 绩效系统 | 数值（原始分） |
| `performance_grade` | 绩效等级 | 绩效系统 | A/B/C/D 映射为 4/3/2/1 |
| `project_contribution` | 项目贡献度 | 项目管理系统 | 1-5 级 |
| `business_value` | 业务价值输出 | 项目成果/OKR | 1-5 级 |

---

### 大类 4：人才标签维度（重要）

| 维度名 | 说明 | 数据来源 | 计分方式 |
|--------|------|----------|----------|
| `is_high_talent` | 是否高层次人才 | 花名册（第6列） | 0/1 |
| `high_talent_type` | 高层次类型 | 花名册 | 架构师/拔尖/金种子 |
| `it_expert_level` | IT专家等级 | 花名册 | L3=1分 L4=2分 无=0分 |
| `tech_stack` | 技术栈 | 花名册/项目经历 | 多值标签 |
| `specialty_direction` | 专业方向 | 花名册 | 枚举 |

---

### 大类 5：发展维度

| 维度名 | 说明 | 数据来源 | 计分方式 |
|--------|------|----------|----------|
| `development_willingness` | 发展意愿 | 调研/访谈 | 1-5 级 |
| `potential_score` | 潜力评估分 | 潜力模型 | 1-5 级 |
| `training_hours` | 年度培训学时 | 九天平台 | 数值 |
| `career_aspiration` | 职业志向 | 调研 | 管理序列/技术序列 |
| `certifications` | 资质证书数 | 培训记录 | 数值 |

### 大类 6：协作维度

| 维度名 | 说明 | 数据来源 | 计分方式 |

## 三、筛选引擎 Skill 设计

### 3.1 核心功能

```
输入（用户提问）：
"我需要一个前端开发，负责一个用户端项目，需要沟通能力强，有创新能力，绩效不能低于B，最好是青年员工"

↓ 解析为：

任务需求：
{
  required_dimensions: [
    { dim: "tech_stack", value: "前端", operator: "contains" },
    { dim: "communication", min: 4, weight: 1.5 },
    { dim: "innovation", min: 3, weight: 1.2 },
    { dim: "performance_grade", min: 2, weight: 1.0 },
    { dim: "is_youth", value: 1, weight: 0.8 }
  ],
  sort_by: "weighted_score",
  limit: 10
}

↓ 筛选引擎计算：

输出：
[
  { name: "张三", weighted_score: 4.7, matched_dimensions: [...], gaps: [...] },
  { name: "李四", weighted_score: 4.3, matched_dimensions: [...], gaps: [...] },
  ...
]
```

### 3.2 用户自定义公式

用户可指定评分公式，支持两种模式：

**模式 A：加权求和（默认）**
```
score = Σ (dimension_score × weight) / Σ weight
```

**模式 B：用户自定义公式**
```
# 用户输入
formula: "(tech_skill × 0.3 + communication × 0.25 + innovation × 0.2 + performance_score × 0.25) / 4"
filters: "tech_stack == '前端' AND performance_grade >= 2 AND is_youth == 1"
sort: "desc"
limit: 10
```

### 3.3 筛选维度参数

```typescript
interface ScreeningRequest {
  // 筛选条件（满足这些才能进入排名）
  filters: {
    dimension: string;       // 维度名
    operator: 'eq' | 'ne' | 'gt' | 'gte' | 'lt' | 'lte' | 'contains' | 'in';
    value: any;
  }[];

  // 排名维度（参与加权计算的维度）
  ranking_dimensions: {
    dimension: string;        // 维度名
    weight?: number;           // 权重（默认1.0）
    // 或使用自定义公式时不需要 weight
  }[];

  // 自定义公式（可选，不填则用加权求和）
  formula?: string;

  // 排序
  sort: 'asc' | 'desc';

  // 返回数量
  limit: number;
}
```

### 3.4 输出格式

```typescript
interface ScreeningResult {
  total_candidates: number;    // 符合筛选条件的总人数
  returned_count: number;       // 返回的数量
  ranking: {
    rank: number;
    employee_id: string;
    name: string;
    group: string;
    overall_score: number;     // 综合得分
    dimension_scores: {       // 各维度分项得分
      dimension: string;
      score: number;
      weight: number;
      weighted_score: number;
    }[];
    matched_filters: string[]; // 满足的筛选条件
    unmet_filters: string[];   // 不满足的筛选条件（警告）
    gaps: string[];           // 与满分差距分析
    recommendation?: string;  // 简单推荐理由
  }[];
  query_time: string;
}
```

### 3.5 筛选示例

**示例1：项目组队**
```
输入：从 IT专家人才池（L4）中选3人，组成一个AI算法攻关小组，要求沟通好、创新强、绩效>=B

解析：
filters: [
  { dimension: "it_expert_level", operator: "eq", value: 2 },  // L4
]
ranking_dimensions: [
  { dimension: "communication", weight: 1.5 },
  { dimension: "innovation", weight: 1.5 },
  { dimension: "performance_grade", weight: 1.0 },
]

输出：排名列表
```

**示例2：培训名额分配**
```
输入：给数字化转型培训班分配10个名额，要求青年员工、绩效>=B、学习意愿高

解析：
filters: [
  { dimension: "is_youth", operator: "eq", value: 1 },
  { dimension: "performance_grade", operator: "gte", value: 2 },
]
ranking_dimensions: [
  { dimension: "learning_ability", weight: 1.5 },
  { dimension: "development_willingness", weight: 1.2 },
  { dimension: "training_hours", weight: 0.8 },
]
```

**示例3：晋升候选人提名**
```
输入：从高级别员工中提名5人进入管理梯队，要求管理潜力>=4、发展意愿>=4、绩效>=A

解析：
filters: [
  { dimension: "position_level", operator: "gte", value: 7 },
  { dimension: "potential_score", operator: "gte", value: 4 },
  { dimension: "development_willingness", operator: "gte", value: 4 },
  { dimension: "performance_grade", operator: "eq", value: 4 },  // A=4
]
ranking_dimensions: [
  { dimension: "management_skill", weight: 2.0 },
  { dimension: "leadership_recognition", weight: 1.5 },
  { dimension: "collab_rating", weight: 1.0 },
]
```

## 四、数据填充策略

### 4.1 数据来源矩阵

| 维度 | 已有数据 | 需补充采集 |
|------|----------|------------|
| 基础属性 | ✅ 花名册（直接可用） | - |
| 岗级/组别 | ✅ 花名册 | - |
| 绩效分 | ⚠️ 绩效系统（需对接） | - |
| 高层次人才标签 | ✅ 花名册第6列 | - |
| IT专家等级 | ✅ 花名册 | - |
| 技术能力 | ❌ 无结构化数据 | 项目经历采集 |
| 管理能力 | ❌ 无数据 | 360评估 |
| 沟通/协作 | ❌ 无数据 | 360评估 |
| 创新/潜力 | ❌ 无数据 | 专项调研 |
| 发展意愿 | ❌ 无数据 | 问卷调研 |
| 培训学时 | ⚠️ 九天平台（需对接） | - |

### 4.2 画像完善优先级

```
第一阶段（直接可用）：
  - 基础属性维度（花名册直接映射）

第二阶段（系统对接）：
  - 绩效维度（绩效系统）
  - 培训学时（九天平台）

第三阶段（需专项采集）：
  - 能力维度（360评估 or 主管评定）
  - 协作维度（360评估）
  - 潜力/发展意愿（问卷调研）
```

## 五、Skill 落地形式

### 5.1 提供两个 Skill

**Skill 1：employee_screening（员工筛选查询）**
- 功能：根据任务需求，从员工池中筛选符合条件的员工并排名
- 输入：任务描述 or 结构化筛选条件
- 输出：排名列表 + 各维度得分 + 差距分析

**Skill 2：employee_profile（员工画像查询）**
- 功能：查询指定员工的完整画像（所有维度得分）
- 输入：员工姓名 or 工号
- 输出：该员工的完整画像报告

### 5.2 使用方式

```
用户：帮我从147人里找出适合做AIPC项目的5个人，要求技术强、沟通好、有创新

→ employee_screening Skill
→ 返回排名列表 + 推荐理由 + 各维度差距

用户：张三的画像是什么样的

→ employee_profile Skill
→ 返回张三的所有维度得分 + 雷达图数据
```

## 六、需要你制定的规则

以下内容需要你来拍板，写入执行计划：

| # | 问题 | 影响因素 |
|---|------|----------|
| 1 | **360评估做不做？** 不做的话能力/协作维度用什么数据替代？ | 能力维度数据来源 |
| 2 | **潜力模型怎么建？** 有没有历史晋升/绩效数据可以训练？ | 发展维度 |
| 3 | **数据更新频率？** 月更还是实时？影响技术架构 | 技术方案 |
| 4 | **花名册里147人的数据完整度如何？** 有没有缺失字段？ | 数据治理工作量 |
| 5 | **评分标准化：** 5分制、10分制、百分制，你倾向哪个？ | 统一口径 |
| 6 | **跨组筛选规则：** 跨组支援时要不要限制组别？ | 筛选过滤条件 |
| 7 | **公式谁来定？** HR定还是部门负责人定？ | 权责 |

## 七、执行计划（初稿）

```
Phase 1：数据盘点 & 维度确认（1周）
  - 盘点147人花名册数据完整度
  - 确定每个维度的数据来源
  - 确认7个拍板问题

Phase 2：数据接入 & 画像构建（2-3周）
  - 花名册数据清洗入库
  - 绩效/培训系统对接
  - 缺失字段补录方案

Phase 3：Skill开发（2周）
  - employee_screening Skill
  - employee_profile Skill
  - 公式配置化（用户可自定义）

Phase 4：试点运行（1-2周）
  - 选一个真实任务场景试用
  - 收集反馈，调整权重/公式

Phase 5：正式上线 & 迭代
  - 补充360评估数据
  - 完善能力/协作维度
```
